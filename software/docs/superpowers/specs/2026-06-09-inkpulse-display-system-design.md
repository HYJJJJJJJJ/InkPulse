# InkPulse 墨水屏显示系统 — 设计文档

- 日期：2026-06-09
- 状态：设计已与用户确认，待评审后进入实现计划
- 范围：`software/` 目录下的全部软件（开发机侧 Hub + 设备侧固件 + Claude Code hook 脚本）

---

## 1. 背景与目标

基于已有硬件（ESP32-S3 + 7.5" 三色墨水屏）开发一个**桌面信息屏**，显示：

- **Claude 状态（A）**：某台开发机上 Claude Code 的实时工作状态
- **Claude 用量（B）**：订阅制下的 token 用量 / 花费估算 / 计划窗口剩余
- **待办（C）**：工作/生活待办事项
- **照片（D）**：照片展示
- **白送项**：板载温湿度 + 时钟/日期

核心诉求：**显示内容可配置**，默认布局由实现方给出合理值。

### 非目标（v1 不做，YAGNI）
- 图形化拖拽布局编辑器（v1 用 YAML 配置）
- 云端中继 / 远程访问（局域网内使用；中继留作后续可选插件）
- 电池供电 / 低功耗优化（硬件常插电，无电池）
- API Key 计费模式的用量（用户是订阅制；详见 §4.2）

## 2. 硬件事实（来自原理图与 E075A42 规格书）

| 模块 | 事实 | 对软件的约束 |
|---|---|---|
| 主控 | ESP32-S3 模组（YD-ESP32-S3），WiFi + BLE，含 PSRAM | 联网无碍；PSRAM 足以放整屏帧缓冲 |
| 屏幕 | E075A42，7.5"，**800×480**，**黑/白/红三色 (BWR)**，控制器 **UC8179**，4 线 SPI；有效区 163.2×97.92mm | 全刷约 15~30s，**无局刷、无灰阶、无动画**；红色仅用于强调/告警 |
| 屏幕信号 | SCL(SPI CLK)、SDA(MOSI)、CSB(CS)、DC、RES(复位)、BUSY(忙信号输入)；BS 选择 3/4 线 SPI；外部升压由 GDR/RESE 驱动板载 MOSFET(SI2304)+电感，芯片内部时序控制 | 固件只需驱动 SPI + 控制脚；上电升压由面板上电命令触发，硬件已就绪 |
| 供电 | USB-C 5V → LDO(CJ6107B33M) → 3.3V，**无电池、无充电** | 常插电桌面设备，可常驻联网、按计划刷新 |
| 传感器 | 板载 HTU21D 温湿度（I2C：IIC_SDA / IIC_SCL） | 时钟/温湿度 widget 数据源 |
| 输入 | 仅 BOOT / RESET 两键，1 个红色 LED | 几乎无物理交互，配置/控制走网络侧 |
| 烧录 | CH343G USB 串口 + DTR/RTS 自动下载 | 调试方便 |

> 帧缓冲大小：800×480 = 384,000 px；1 bit/px/plane = 48,000 B；BWR 双 plane（黑 + 红）= 96,000 B ≈ 94 KB，PSRAM 充裕。

> ESP32 与屏幕/传感器的具体 GPIO 映射以 PCB 网表为准，bring-up 阶段核对。

## 3. 总体架构（方案：瘦客户端 + 本机 Hub 渲染）

```
开发机 (Claude Code 订阅)                          墨水屏设备 (ESP32-S3)
┌──────────────────────────────────────┐
│  Claude Code ──hooks──┐               │
│                       ▼               │
│  ┌──────────  InkPulse Hub  ──────────┐│   WiFi / HTTP     ┌────────────────┐
│  │ 采集器 → 数据模型 → 渲染引擎        ││───────────────────▶│ 固件:           │
│  │                    → HTTP /frame    ││  位图 + ETag       │  拉帧→变了才刷  │
│  │                    → Web /todos     ││  + X-Next-Refresh  │  →轻睡→循环     │
│  └─────────────────────────────────────┘│                    │  离线缓存上一帧│
│   ▲config.yaml  ▲photos/  ▲todo 存储     │                    └────────────────┘
└──────────────────────────────────────┘
```

**渲染在电脑上完成**：Hub 把整屏渲染成三色位图，设备只负责"拉下来贴上去"。改布局只改配置、不烧固件；三色照片抖动也只能在电脑上做好。

**离线处理（内建，与开发机是否常开无关）**：设备连不上 Hub 时保留上一帧并显示离线角标，退避重连。

## 4. 组件设计（边界清晰，各自可独立测试）

### 4.1 Hook 脚本（Claude Code 侧）
- 挂在 Claude Code 的 `settings.json` hooks：`SessionStart` / `Stop` / `Notification` / `UserPromptSubmit`（及可用的工作开始/结束钩子）
- 每次触发：把当前状态 + 项目名 POST 到 Hub（如 `POST /ingest/claude-status`）
- 脚本保持极薄；网络失败静默忽略（不阻塞 Claude Code）

### 4.2 采集器（Collectors，Hub 内，各产出一个类型化数据模型）
1. **Claude 状态采集器**：接收 hook POST，维护状态机
   `idle / working / waiting_for_input / done(刚完成) / error`，附带 `project`、`since`（进入该状态时间）
2. **用量采集器**：解析订阅会话日志 `~/.claude/projects/**/*.jsonl`（ccusage 思路）
   产出：今日 token（输入/输出/缓存）、估算花费、Max 计划 5 小时窗口剩余比例、今日会话数
   > 订阅制无官方用量 API，只能解析本机日志；日志结构变更需容错（缺失 → 该项 `n/a`）
3. **待办采集器**：读取待办存储（见 §4.5）
4. **照片采集器**：扫描照片文件夹，按策略选/轮换一张

每个采集器：输入明确（hook / 日志文件 / 存储 / 文件夹），输出一个 dataclass，依赖隔离，可单测。

### 4.3 渲染引擎（Hub 内）
- 输入：`config.yaml`（布局+启用的 widget）+ 各数据模型
- 流程：按 widget 把内容排版到 800×480 画布 → 量化为 **黑 plane + 红 plane** 两个 1-bit 位图
  - 文本/图标：纯黑白 + 红色强调
  - 照片：**Floyd–Steinberg 三色抖动**（黑/白/红）
- 技术：Python + Pillow
- 产物：两个 bitplane 的字节缓冲 + 一个内容哈希（用于 ETag）

### 4.4 HTTP 服务（Hub 内）
- `GET /frame`：返回打包的双 plane 位图；响应头 `ETag`（内容哈希）、`X-Next-Refresh`（建议下次拉取间隔秒数）；请求带 `If-None-Match` 命中则 `304`
- `GET /preview.png`：把当前帧渲染成 PNG，**在电脑上肉眼调布局**用（无需设备）
- `GET /health`：健康检查
- `POST /ingest/claude-status`：接收 hook 上报
- `GET|POST /todos` + 静态页：待办 Web UI（见 §4.5）
- 技术：FastAPI

### 4.5 待办 Web UI（按用户要求新增）
- Hub 提供一个轻量网页 `/todos`：增加 / 勾选完成 / 删除待办
- 局域网可访问 → 手机也能加待办
- 后端存储：简单持久化（JSON 或 SQLite，实现时定；二选一即可）
- Markdown 文件作为**可选导入源**（`- [ ]` 语法），非主路径

### 4.6 固件（设备侧，ESP-IDF）
- **WiFi STA**：连接配置好的 AP
- **配网**：v1 用 SoftAP 配网页一次性写入 NVS（无按键也能配）；实现复杂时先编译期写死兜底
- **主循环**：
  1. 连 WiFi（断则重连退避）
  2. `esp_http_client` 请求 `GET /frame`，带 `If-None-Match: <上次ETag>`
  3. `304` → 不刷屏；`200` → 解析双 plane → 写 UC8179 → 全刷
  4. 按响应头 `X-Next-Refresh` 延时 / 轻睡，循环
- **墨水屏驱动**：ESP-IDF 自写精简 **UC8179 SPI 驱动**（命令表取自 E075A42 规格书）：复位时序、上电、写黑 RAM、写红 RAM、刷新、深睡
- **离线**：保留上一帧（NVS/RAM 缓存最近 ETag 与帧），连不上时维持显示 + 角标
- **温湿度**：设备 I2C 读 HTU21D，在请求 `/frame` 时把读数随请求带上（query/header），由 Hub 渲染进顶栏（数据归属明确：设备采集、Hub 渲染）
- 不上 LVGL：瘦客户端只需整屏 blit 双 plane

## 5. 设备 ⇄ Hub 协议

| 项 | 约定 |
|---|---|
| 取帧 | `GET /frame`，`Accept` 二进制；带 `If-None-Match` |
| 帧格式 | 头部(宽/高/plane数/字节序) + 黑 plane(48000B) + 红 plane(48000B)；可选 RLE 压缩 |
| 去重 | `ETag` = 帧内容哈希；未变返回 `304`，设备不刷屏（**三色屏体验命门：避免无谓闪烁**） |
| 节奏 | 响应头 `X-Next-Refresh` 由 Hub 决定下次拉取间隔（秒） |
| 上报 | 设备可在取帧时附带温湿度（query 或 header） |

## 6. 默认布局（实现方决定，可配置覆盖）— 组合仪表盘，不翻页

> 翻页每切一次全刷（闪 15~30s），故默认单屏组合：

```
┌──────────────────────────────────────────────┐
│ 6/9 周一 14:32        🌡22°C 💧55%   [顶栏]     │
├───────────────────────────┬────────────────────┤
│   🔴 等你输入              │ 今日用量            │
│   project: InkPulse        │  1.2M tok ≈ $0      │
│   (Claude 状态 — 主角)     │  5h窗口 ▓▓▓▓░ 68%   │
├───────────────────────────┴────────────────────┤
│ ☐ 完成固件刷屏  ☐ 写 todos 解析  ☑ 定架构        │
│ (待办 取前几条)                                  │
└──────────────────────────────────────────────┘
```

- 红色用于"等你输入 / 出错"等需注意状态
- **照片**作为**可选布局**（如配置成 "Claude 空闲时整屏照片" 或定时切换），不挤默认仪表盘

## 7. 刷新策略（针对三色屏慢刷）

- **状态变化事件驱动**刷新，设**最小刷新间隔**（如 ≥60s）+ 防抖，避免连续闪屏
- **用量/时钟定时**刷新（如每 5~10 分钟）；时钟只到分钟，不显示秒
- **照片**慢速轮换（如 30~60 分钟）
- **ETag 去重**：帧未变设备不刷屏
- 调度逻辑集中在 Hub（设备只听 `X-Next-Refresh`），便于调参

## 9. 容错

- **Hub**：任一采集器失败 → 对应 widget 显示 `n/a`，不拖垮整帧；渲染按 widget 隔离
- **设备**：连不上 Hub → 留上一帧 + 离线角标 + 退避重连；WiFi 断 → 重连循环
- **Hook**：上报失败静默，不影响 Claude Code

## 11. 落地分期（详细步骤留给实现计划）

- **Phase 1 — 打通管道**：Hub 渲染一张**静态**仪表盘 → `/frame` → 固件拉取并全刷成功（含 ETag/304、离线缓存、UC8179 驱动）
- **Phase 2 — 接真实数据**：Claude 状态 hook + 用量日志解析 → 仪表盘活起来
- **Phase 3 — 待办/照片/配置化**：待办 Web UI + 照片抖动 + `config.yaml` 驱动布局 + 刷新策略调参
